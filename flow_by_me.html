Mini Exception Inbox

Project Walkthrough & Technical Decisions

By Vishal Raj

1. Demo of the App Working End-to-End

The application is completely containerized. By running a single command, docker compose up --build, the entire ecosystem spins up automatically.

Once started, the backend (Node.js/Express) automatically processes the initial CSV data files through our ETL pipeline, populating the database with raw records, cleaning them, and generating the necessary production exceptions.

The User Experience:

  • Dashboard view: Planners navigate to the React frontend (running at port 5173). They see a timeline-based inbox where exceptions are grouped by day.
  • Filtering: Users can filter exceptions by Severity (High, Medium) or by specific Product codes.
  • Interaction: Clicking on an exception row slides out a detailed drawer panel. This panel displays the planned vs. actual numbers, the deficit percentage, and an interactive Recharts-powered 7-day trend chart.
  • Action: Planners can change the status of an exception from OPEN to ACKNOWLEDGED or RESOLVED directly from the drawer. This updates the backend via a REST API and refreshes the UI instantly without a page reload using Redux Toolkit.

2. Database Schema & Key Decisions

I decided to use SQLite with Prisma ORM to keep the setup zero-configuration and lightweight as per the assignment constraints, instead of defaulting to a heavier database like PostgreSQL or MongoDB.

Instead of simply having two tables, I designed the database into three logical layers to mimic an enterprise Data Warehouse / ERP structure:

The Layers:

  1. Raw Tables (raw_production_plan & raw_actual_production):
    Stores the exact, unmodified data from the CSV files. This is crucial for auditing imports and allows us to replay the ETL process if business logic changes.
  2. Clean Tables (production_plan & actual_production):
    This contains the normalized data after validation and transformation.
  3. Materialized View (exceptions):
    Instead of calculating deficits dynamically on every API request (which gets slow with large datasets), I materialized the exceptions into their own table. It stores fields like planned_units, actual_units, deficit_units, severity, and status.

Key Decision: Materializing Exceptions

Calculating the exceptions on the fly via SQL JOINs on every page load would negatively impact read performance on the frontend. By executing the calculation logic once during the Load phase of the ETL pipeline and storing it in the exceptions table, the frontend API endpoints just perform simple SELECT queries. This drastically speeds up the dashboard.

3. My Process: ETL Pipeline & Data Discoveries

The data ingestion process follows a strict Extract, Transform, Load (ETL) pattern handled by custom scripts in the backend.

What I Found in the Data

During the Extract phase, my validation script caught a data quality issue in production_plan.csv: two records contained missing values for planned_units.

Instead of letting the application crash or inserting corrupted `NULL` data into the system, I implemented a fault-tolerant import process:

  • Every row is strictly validated before insertion.
  • The script logged the specific invalid row numbers.
  • The 2 corrupted rows were skipped, while the other 1,083 valid rows were successfully imported.

Tradeoffs Made

  • SQLite vs PostgreSQL: I used SQLite because the assignment required a lightweight, easy-to-run setup. The tradeoff is that SQLite handles concurrent writes poorly compared to Postgres, but for an isolated dashboard, it is more than sufficient.
  • Batch Processing vs Real-time: The ETL script processes CSVs in batches. In a true enterprise scenario, we might use message queues (like Kafka or RabbitMQ) to ingest production updates in real-time, but for this scope, a batch script was a pragmatic tradeoff.
  • AI Assistance vs Hand-Coding: I heavily utilized AI (GPT-5.5) to scaffold the Prisma boilerplate and UI layouts, allowing me to focus roughly 30% of my time entirely on architectural design, data validation logic, and complex state management, rather than typing out basic CRUD endpoints.

4. What I Would Do Differently (Future Improvements)

Given more development time, I would elevate the project from a solid prototype to a production-ready application by implementing the following:

  • Background ETL Workers: Move the CSV parsing and data ingestion out of blocking scripts and into background jobs using tools like BullMQ or Redis.
  • Authentication & Authorization: Implement JWT-based login so different planners can have their own accounts, and we can track who resolved an exception in an Audit Log.
  • Real-time UI Updates: Integrate WebSockets (Socket.io) so that if one planner acknowledges an exception, all other logged-in planners see the status update instantly without needing to refresh their dashboards.
  • Automated Testing: Add a comprehensive suite of unit and integration tests using Jest and Supertest to ensure the ETL logic never regresses.
  • Server-Side Caching: Use Redis to cache the dashboard statistics, reducing database hits on the initial dashboard load.
